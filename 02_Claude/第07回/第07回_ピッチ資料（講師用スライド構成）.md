### B. 設計の考え方（Slide 04〜07｜05〜15分）

#### Slide 04｜AI業務担当者とは何か

**タイトル：** AI業務担当者 ＝ 人間が責任を持って指示・確認する業務支援の仕組み

**大見出し：**
> 「新人に業務を引き継ぐ」のと同じ感覚で、AIに業務を任せる設計図を作る

**構成要素カード（6枚・横2×縦3）：**

| 要素 | 説明 | 例 |
|---|---|---|
| **役割** | 何の担当か | 配送実績レポート担当 |
| **参照情報** | 業務に必要な知識・過去データ | 配送エリアマスター、ドライバー名簿 |
| **業務手順（スキル）** | Phase①で作ったスキル＋新規スキル | 配送実績レポート作成スキル |
| **入力・出力** | 何を受け取り、何を生成するか | CSV受け取り → Excel＋メール文面 |
| **実行頻度** | いつ・どのくらいの頻度で動くか | 毎朝9時 |
| **権限・承認ポイント** | AIがやってよい範囲、人間が確認する箇所 | ファイル作成OK・送信は人間が確認 |

**補足テキスト：**
> 第06回で体験した「プロジェクト＋スキル＋スケジュール」を、**自分の業務向けに設計する**のが今日の作業です。

> 💡 **講師メモ（ノート欄）：**  
> 「AI業務担当者は"AIが自律的に判断する"仕組みではありません。"人間が責任を持って指示・確認する業務支援の仕組み"です。新人に業務を引き継ぐときと同じように、役割・手順・権限を明確にします。」

---

#### Slide 05｜設計シートの全11項目

**タイトル：** AI業務担当者設計シート ── 11の設計項目

**項目リスト（縦並び・番号付き）：**

| No. | 項目 | 記入のポイント |
|---|---|---|
| ① | **担当者名** | わかりやすい名前（例：AI配車アシスタント） |
| ② | **役割** | 何の業務を担当するか（1〜2文で） |
| ③ | **参照情報** | 業務知識・過去データ・マニュアル・マスターデータ |
| ④ | **使用スキル** | Phase①で作ったスキル＋新規で必要なスキル |
| ⑤ | **入力トリガー** | 何をきっかけに動き出すか（ファイル受領・時刻・人の指示） |
| ⑥ | **出力形式** | 生成するファイルの種類・形式・保存先 |
| ⑦ | **実行頻度** | 毎日・毎週・毎月・都度 |
| ⑧ | **権限（AIがやってよい範囲）** | ファイル作成、集計、文面生成 等 |
| ⑨ | **承認ポイント（人間が確認すべき箇所）** | 送信前・異常値発生時・金額計算後 等 |
| ⑩ | **禁止事項** | やってはいけないこと（データ外部送信・勝手な送信 等） |
| ⑪ | **テスト方法** | 第三者が運用マニュアルだけで実行できるか |

**デザイン提案：** ⑧⑨⑩をオレンジ枠で囲み「安全設計の3項目」として強調。

**重要メッセージ（オレンジ枠）：**
> ⚠️ ⑧権限・⑨承認ポイント・⑩禁止事項 ── この3つが「安全な業務適用」の基盤です

> 💡 **講師メモ（ノート欄）：**  
> 「11項目ありますが、全部を完璧に書く必要はありません。まず①〜⑦を埋めてから、⑧⑨⑩の安全設計に取り組む流れで進めてください。⑪のテスト方法は第08回で実際にテストするので、今日は方針だけでOKです。」

#### Slide 06｜記入例：AI配車アシスタント

**タイトル：** 記入例 ── AI配車アシスタント（物流企業）

**設計シート記入例（テーブル）：**

| 項目 | 記入内容 |
|---|---|
| **① 担当者名** | AI配車アシスタント |
| **② 役割** | 日次配送実績の集計・レポート生成を担当する |
| **③ 参照情報** | 配送エリアマスター、ドライバー名簿、過去3ヶ月の実績データ |
| **④ 使用スキル** | 配送実績レポート作成スキル（Phase①で作成済み） |
| **⑤ 入力トリガー** | 毎朝9時に前日の配送実績CSVが所定フォルダに保存されたとき |
| **⑥ 出力形式** | Excel（配送先別集計表＋グラフ）＋上司向けサマリーメール文面（Word） |
| **⑦ 実行頻度** | 毎日（平日のみ） |
| **⑧ 権限** | レポートファイルの作成はOK。メール文面の生成はOK |
| **⑨ 承認ポイント** | ①メール送信は人間が内容確認後に実行 ②異常値（前日比20%以上の変動）があった場合は人間に報告して判断を仰ぐ |
| **⑩ 禁止事項** | 顧客情報の外部送信、メールの自動送信、データの削除 |
| **⑪ テスト方法** | 同僚にマニュアルを渡し、CSVを指定フォルダに置いてレポートが正しく生成されるか確認してもらう |

**デザイン提案：** テーブルを全面表示。⑧⑨⑩の行をオレンジの薄い背景色でハイライト。

> 💡 **講師メモ（ノート欄）：**  
> 「この例を見てください。ポイントは⑨の承認ポイントです。"メール送信は人間が確認してから"、"異常値は人間に報告"と書いてあります。AIに全部任せるのではなく、『ここは人間が判断する』ラインを明確にするのが設計のコツです。⑧の権限と⑩の禁止事項はセットで考えてください。」

#### Slide 07｜自社にどんなAI業務担当者が必要か考えるフレームワーク

**タイトル：** 「どんなAI業務担当者を作るか」を決めるフレームワーク

**3つの問い（大きなカード形式）：**

| 問い | 考えるヒント |
|---|---|
| **Q1. どの業務が一番「毎回同じことをしている」か？** | 第01回で棚卸した定型業務の中から、Phase①のスキルの延長で設計できるもの |
| **Q2. その業務で「AIにここまではやってほしい、ここからは自分で確認したい」ラインはどこか？** | 作成はAI、確認・送信・判断は人間 → この境界線が⑧⑨⑩になる |
| **Q3. 明日から第三者が代わりに実行できるか？** | Yesなら設計が十分。Noなら手順が曖昧な箇所がある |

**接続：Phase①のスキルとの関係**

```
Phase①で作ったスキル（手順書）
    ↓ これに以下を追加する
役割 ＋ 参照情報 ＋ 実行頻度 ＋ 権限・承認ポイント
    ↓
＝ AI業務担当者（プロジェクトとして構築可能な設計図）
```

**補足テキスト：**
> Phase①のスキルは「手順」。AI業務担当者はそのスキルに**役割・知識・権限を加えたもの**です。

> 💡 **講師メモ（ノート欄）：**  
> 「Phase①で作ったスキルを持っている方は、それをベースに設計してください。新しいスキルが必要な方は④に『新規作成が必要』と書いておけばOKです。Q2の『どこまでAIにやらせて、どこから人間が確認するか』が今日の最重要テーマです。これが曖昧だと、業務に使うときに不安が残ります。」

#### Slide 09｜Step 1 ── 基本設計（①〜⑦）のヒント

#### Slide 10｜Step 2 ── 安全設計（⑧⑨⑩）のヒント

**タイトル：** Step 2：安全設計 ── ⑧権限・⑨承認ポイント・⑩禁止事項

**大見出し：**
> 「AIにどこまで任せるか」の境界線を引く ── これが安全な業務適用の基盤

**考え方フレーム：**

```
┌──────────────────────────────────┐
│       AIがやってよいこと（⑧権限）      │
│  ファイル作成・集計・文面生成・整理      │
├──────────────────────────────────┤
│   人間が確認してから進めること（⑨承認）   │
│  送信・公開・上司報告・金額確定・削除    │
├──────────────────────────────────┤
│     絶対にやってはいけないこと（⑩禁止）   │
│  外部送信・個人情報出力・自動送信       │
└──────────────────────────────────┘
```

**承認ポイントの典型パターン（テーブル）：**

| 業務場面 | 承認ポイントの例 |
|---|---|
| レポート生成 | 異常値があったら人間に報告 |
| メール・文書の送信 | 送信前に内容を人間が確認 |
| 金額・数量の計算 | 計算結果を人間が検算してから確定 |
| データの更新・削除 | 実行前に変更内容を提示し承認を得る |
| 社外向け資料の作成 | 公開前に人間がチェック |

> 💡 **講師メモ（ノート欄）：**  
> 「ここが今日の一番大事なパートです。⑧⑨⑩は必ずセットで考えてください。『AIにやらせていいこと（権限）』を書いたら、その裏返しとして『ここは人間が確認すること（承認ポイント）』と『絶対ダメなこと（禁止事項）』が決まります。迷ったら"万が一ミスしたときに影響が大きい箇所"を承認ポイントにしてください。」

#### Slide 12｜講師巡回ポイント（講師用・投影不要）

**タイトル：** 【講師用】巡回時のチェックポイント

**巡回ポイント（5項目）：**

| 受講者の状態 | 講師の対応 |
|---|---|
| 「何をAI業務担当者にすればいいかわからない」 | 「Phase①で作ったスキルはありますか？そのスキルに、毎回使う知識やデータを足したものがAI業務担当者です」 |
| 「権限の範囲がわからない」 | 「その業務でミスしたとき、影響が小さいものはAIに任せてOK。影響が大きいものは人間確認にしましょう」 |
| 「承認ポイントが思いつかない」 | 「送信・公開・確定のタイミングで人間が確認する、で十分です。異常値の発見も入れましょう」 |
| 「参照情報が何かわからない」 | 「その業務をやるとき、いつもどのファイルやデータを見ていますか？それが参照情報です」 |
| 「Step 1で止まっている」 | 「まず②の役割だけ書いてください。"○○のレポートを作る担当"でOKです。そこから広げましょう」 |

**巡回タイミングの目安：**
> ① 開始5分後：全員が作業に入れているか確認  
> ② 開始10分後：Step 1（基本設計）の進捗確認  
> ③ 開始20分後：Step 2（安全設計）に入れているか確認  
> ④ 開始30分後：⑧⑨⑩が埋まっているか確認、ペアレビューの準備アナウンス

> 💡 **講師メモ（ノート欄）：**  
> このスライドは投影不要。講師の手元資料として使用。最も詰まりやすいのは③参照情報と⑧⑨⑩の安全設計。「新人に引き継ぐとしたら？」「ミスしたら誰に怒られる？」が魔法の問いかけ。

#### Slide 13｜設計が詰まったときの追加ヒント

**タイトル：** 手が止まったら ── 「業務の流れ」を先に書き出す

**アプローチ：**
> 設計シートが埋まらないときは、**まず今やっている業務の流れを箇条書き**にしてみましょう。

**書き出しテンプレート：**

```
自分の業務の流れ（現状）：

1. _________ がきっかけで作業を始める → ⑤入力トリガー
2. _________ のデータ・ファイルを開く → ③参照情報
3. _________ の作業をする → ④使用スキル
4. _________ ができたら完了 → ⑥出力形式
5. _________ に送る／報告する → ⑨承認ポイント候補
```

**補足テキスト：**
> この流れを書いてから設計シートに転記するとスムーズです。  
> 「仕事の流れを書き出してからスキルに落とす」方法は、AI業務担当者の設計でも有効です。

> 💡 **講師メモ（ノート欄）：**  
> このスライドは全員に見せる必要はない。巡回時に手が止まっている受講者に対して「まず業務の流れを5行で書いてみましょう」と声をかけ、必要に応じてこのスライドを見せる。ref7（野島さんの事例）と同じ「仕事の流れを書き出してからスキルに落とす」アプローチ。

### D. ペアレビュー（Slide 14〜15｜50〜55分）

---

#### Slide 14｜ペアレビュー

**タイトル：** 隣の人と設計を共有しよう（5分）

**やること（2ステップ）：**

| Step | 内容 | 時間 |
|---|---|---|
| ① | 自分の設計シートを見せながら「何のAI業務担当者を設計したか」を説明する | 1人2分 |
| ② | 相手の設計に対してフィードバックを1つずつ伝える | 各30秒 |

**レビュー観点（3つ・大きく表示）：**

| 観点 | チェックポイント |
|---|---|
| 🤔 **第三者が見てわかるか** | 「この設計書だけ読んで、私がそのAI業務担当者を動かせる？」 |
| 🔒 **承認ポイントは適切か** | 「ここは人間が確認しなくて本当に大丈夫？」 |
| ⚠️ **禁止事項に漏れはないか** | 「万が一AIが暴走したとき、何が起きると困る？」 |

**進行目安：** 1ペア5分（2分×2人 + フィードバック1分）

> 💡 **講師メモ（ノート欄）：**  
> 「2人1組で設計シートを見せ合ってください。特に重要なのは2つ目の観点です。『承認ポイントは適切か』、つまり『ここは人間が確認しなくて大丈夫？』という視点でチェックしてください。自分では気づかない"確認すべきポイント"を相手が見つけてくれることが多いです。」
