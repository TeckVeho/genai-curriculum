#### Slide 04｜業務適用設計シートの全8項目

**タイトル：** 業務適用設計シート ── 8つの設計項目

**項目リスト（縦並び・番号付き）：**

| No. | 項目 | 記入のポイント |
|---|---|---|
| ① | **業務名** | 何の業務か（1〜2文で。具体的に） |
| ② | **現状フロー** | 今の手動の工程を順番に書き出す（5〜8ステップ） |

**重要メッセージ：**
> まず②③を書く。**仕事の流れを書き出してからAIに任せる部分を決める**のが設計の基本です。

> 💡 **講師メモ（ノート欄）：**  
> 「8項目ありますが、全部を完璧に書く必要はありません。まず②の現状フローを書き出してください。仕事の流れ全体を見渡してから、どこをAIに任せるかを決める。この順番が大事です。いきなり③から書き始めると、人間が判断すべき箇所を見落としがちです。YouTube参考動画（ref7）でも強調されていた『仕事の流れを書き出してからスキルに落とす』というアプローチです。」

#### Slide 05｜記入例：配送実績→異常検出→報告→請求更新ワークフロー

**タイトル：** 記入例 ── 配送実績レポート→異常検出→報告→請求更新（物流企業）

**設計シート記入例（テーブル）：**

| 項目 | 記入内容 |
|---|---|
| **① 業務名** | 配送実績の日次処理（実績集計→異常検出→上司報告→請求データ更新） |
| **② 現状フロー** | ❶基幹システムから配送実績CSVをダウンロード → ❷Excelで配送先別に集計 → ❸前日比で異常値（20%以上の変動）を目視チェック → ❹異常があれば上司にメール報告 → ❺問題なければ請求システムの金額を手入力で更新 → ❻更新結果を共有フォルダに保存 |
| **③ AIフロー** | ❶🤖CSVを読み込み自動集計 → ❷🤖前日比の変動を自動検出 → ❸🧑異常値の報告内容を人間が確認 → ❹🧑請求更新の金額を人間が検算・承認 → ❺🤖承認後に更新用ファイルを自動生成 → ❻🤖共有フォルダに保存 |
| **④ 承認ゲート** | ・異常値の報告メール送信前に人間が内容確認 ・請求金額の更新前に人間が検算して承認 |
| **⑤ 例外処理** | ・CSVが届かない場合 → 担当者に連絡し、手動取得後に再実行 ・列構成が変わった場合 → エラーログを出して停止、人間が確認 |
| **⑥ 実データ移行計画** | Step 1：ダミーCSV（10件）で動作確認 → Step 2：先月の実データ（マスキング済み）でテスト → Step 3：当日の実データで本番運用開始 |
| **⑦ 運用ルール** | 毎朝9:30に実行 / 担当：配車チーム山田さん / 引継ぎ：指示文テンプレート＋入力ファイルの場所メモを共有フォルダに格納 |
| **⑧ Before/After指標** | 作業時間：Before 90分/日 → After 目標20分/日 / エラー率：Before 月3件の入力ミス → After 目標0件 / 測定方法：1週間の実測記録 |

**デザイン提案：** テーブルを全面表示。③のAIフロー内の🤖（AI担当）と🧑（人間担当）を色分け（AI＝ネイビー、人間＝オレンジ）。④⑤の行を薄いオレンジ背景でハイライト。

> 💡 **講師メモ（ノート欄）：**  
> 「この記入例を見てください。ポイントは②→③の流れです。まず現状の手動フロー6ステップを書き出し、それぞれをAIが担当するか人間が判断するかを分けています。全部AIに丸投げするのではなく、異常値の確認と請求金額の検算は人間がやる。この『丸投げしない』設計が実務で信頼される基盤になります。」

#### Slide 06｜実データへの移行 ── いきなり本番データを使わない

**タイトル：** 実データ移行の3ステップ ── 段階的に安全に進める

**大見出し：**
> ダミーで動かす → テストデータで確認 → 本番データで運用開始

**3ステップ図（横フロー）：**

```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│  Step 1         │    │  Step 2         │    │  Step 3         │
│  ダミーデータ     │ →  │  テストデータ     │ →  │  本番データ       │
│                 │    │                 │    │                 │
│ ・架空の10件程度  │    │ ・先月の実データ   │    │ ・当日の実データ   │
│ ・動作確認が目的  │    │ ・個人情報をマスク  │    │ ・運用ルールに沿って│
│ ・失敗してもOK   │    │ ・結果を検算する   │    │ ・Before/Afterを測定│
└─────────────────┘    └─────────────────┘    └─────────────────┘
```

**各ステップの確認項目：**

| Step | 確認すること | 合格基準 |
|---|---|---|
| **Step 1** ダミーデータ | ワークフローが最後まで動くか | エラーなく最終出力が生成される |
| **Step 2** テストデータ | 実データの構造・量で正しく処理されるか | 結果を手動計算と照合して一致する |
| **Step 3** 本番データ | 運用ルール通りに回せるか | 1週間の実測でBefore/Afterを確認 |

**重要メッセージ（オレンジ枠）：**
> ⚠️ いきなり本番データで動かさない。**Step 1で失敗を安全に経験し、Step 2で精度を確認してからStep 3に進む。**

**マスキングの考え方：**
> 個人情報（顧客名・住所・電話番号等）が含まれる場合は、テスト用にダミー値に置換してから使う。社内のAI利用ポリシーも確認すること。

> 💡 **講師メモ（ノート欄）：**  
> 「実データへの移行は3ステップで進めます。今日のワークではStep 1のダミーデータを使って試作に着手します。次回の検証回でStep 2のテストデータまで進めます。大事なのは『いきなり本番データを使わない』ことです。ダミーで失敗を経験しておくと、本番で慌てません。個人情報のマスキングについては、社内のAI利用ポリシーに従ってください。」

#### Slide 07｜Before/After効果指標の設定方法

**タイトル：** Before/After指標 ── 「何がどれだけ変わるか」を数字で定義する

**大見出し：**
> 数字がないと「便利になった気がする」で終わる。数字があると「月10時間削減」と言い切れる。

**よく使う指標（4カテゴリ）：**

| カテゴリ | 指標の例 | 測定方法 |
|---|---|---|
| ⏱️ **作業時間** | 1回あたりの作業時間、月間の合計作業時間 | ストップウォッチ or 作業ログで計測 |
| ❌ **エラー率** | 入力ミスの件数、手戻りの回数 | 1週間〜1ヶ月の実績を記録 |
| 📊 **処理件数** | 1日に処理できる件数、バッチ処理のサイズ | 導入前後で同一期間を比較 |
| 😊 **定性評価** | 担当者の負担感、引継ぎのしやすさ | 5段階の自己評価 |

**記入のテンプレート：**

| | Before（現状） | After（目標） | 測定方法 |
|---|---|---|---|
| 作業時間 | 例：90分/日 | 例：20分/日 | 1週間の実測記録 |
| エラー率 | 例：月3件のミス | 例：月0件 | エラーログを記録 |
| 処理件数 | 例：1日20件が限界 | 例：1日50件 | 処理ログで比較 |

**重要ポイント（オレンジ枠）：**
> 📌 Beforeの数字は**今週中に測っておく**。導入前の実績がないと、効果を証明できない。

> 💡 **講師メモ（ノート欄）：**  
> 「Before/After指標は第12回の発表で使います。一番大事なのは、Beforeの数字を今のうちに測っておくことです。『だいたい1時間くらい』ではなく、『今週月曜日に実測したら87分だった』と言えると説得力が段違いです。全部の指標を測る必要はありません。作業時間だけでも十分です。」

#### Slide 09｜Step 1 ── フロー設計（①②③）のヒント

**タイトル：** Step 1：フロー設計 ── 「仕事の流れを書き出してからAIに任せる部分を決める」

**現状フロー（②）の書き方ヒント：**

| 問いかけ | 記入例 |
|---|---|
| 🔍 **この業務は何をきっかけに始まる？** | 「毎朝、基幹システムからCSVをダウンロードすることから始まる」 |
| 📂 **どんなファイル・データを使う？** | 「配送実績CSV、配送エリアマスター、ドライバー名簿」 |
| ✏️ **どんな加工・判断をしている？** | 「Excelで集計し、前日比20%以上の変動を目視チェック」 |
| 📤 **最終的に何を・誰に・どう渡す？** | 「集計レポートを上司にメール、請求データをシステムに入力」 |

**AIフロー（③）の書き方ヒント：**

```
現状フローの各ステップに 🤖 または 🧑 をつける：

❶ CSVダウンロード → 🧑（人間がシステム操作）
❷ 配送先別に集計  → 🤖（AIが自動処理）
❸ 異常値チェック   → 🤖（AIが検出）＋ 🧑（人間が判断）
❹ 上司へ報告      → 🤖（文面生成）＋ 🧑（人間が送信）
❺ 請求データ更新   → 🧑（人間が検算・承認）
❻ 結果を保存      → 🤖（AIが自動保存）
```

**補足テキスト：**
> 「丸投げ」ではなく、**自分の判断軸を持つ**ことが大事。「AIがやること」と「人間が判断すること」を明確に分けましょう。

> 💡 **講師メモ（ノート欄）：**  
> 「②の現状フローが書けない方がいたら、4つの問いかけを使ってください。『何をきっかけに始まる？』→『何のデータを使う？』→『どんな加工をしている？』→『最終的に何を渡す？』。この4つに答えるだけでフローが出てきます。③は、②で書いた各ステップにAIマークか人間マークをつけるだけです。判断に迷う工程は、とりあえず人間マークにしておけば安全です。」

---

#### Slide 10｜Step 2 ── 安全設計と例外処理（④⑤）のヒント

**タイトル：** Step 2-a：安全設計 ── 承認ゲートと例外処理を決める

**承認ゲート（④）の考え方：**

| 「このタイミングで人間が確認する」パターン | 例 |
|---|---|
| **送信・公開の直前** | メール送信前、レポート公開前 |
| **金額・数量が確定する前** | 請求金額の更新前、発注数の確定前 |
| **異常値が検出されたとき** | 前日比20%以上の変動、データ欠損の検知 |
| **社外に情報が出るとき** | 顧客向け資料の生成後、外部システムへの連携前 |

**例外処理（⑤）の考え方 ── 2つの問いで導出：**

```
┌──────────────────────────────────────────┐
│  ❶「いつもの入力が ないとき どうする？」      │
│     例：CSVが届かない → 担当者に連絡して手動取得 │
├──────────────────────────────────────────┤
│  ❷「入力が いつもと違うとき どうする？」       │
│     例：列が増えた → エラーログを出して停止     │
└──────────────────────────────────────────┘
```

**重要メッセージ（オレンジ枠）：**
> ⚠️ 例外は2つ以上書いてください。上の2つの問いに答えれば、最低2つは出てきます。

> 💡 **講師メモ（ノート欄）：**  
> 「④の承認ゲートは、③のAIフローで🧑マークをつけた箇所をそのまま転記すればOKです。⑤の例外処理は、2つの問いかけで考えてください。『いつもの入力がないときどうする？』と『入力がいつもと違うときどうする？』。この2つに答えるだけで、例外パターンが出てきます。例外を想定しておくだけで、現場からの信頼度が大きく変わります。」

#### Slide 11｜Step 2 ── 移行計画・運用ルール・指標（⑥⑦⑧）のヒント

**タイトル：** Step 2-b：実データ移行計画・運用ルール・Before/After指標

**⑥ 実データ移行計画のテンプレート：**

| ステージ | 使用データ | 確認すること |
|---|---|---|
| Step 1 | ダミーデータ（__件） | ワークフローが最後まで動くか |
| Step 2 | テストデータ（____のデータをマスキング） | 実データの構造・量で正しく処理されるか |
| Step 3 | 本番データ | 運用ルール通りに回せるか、Before/Afterを測定 |

**⑦ 運用ルールの記入ヒント：**

| 項目 | 問いかけ |
|---|---|
| 実行頻度 | 「毎日？毎週？月次？不定期？」 |
| 担当者 | 「誰が実行する？休みのときは？」 |
| 引継ぎ方法 | 「他の人に渡すとき、何を共有する？」 |
| 保存場所 | 「生成物はどこに保存する？」 |

**⑧ Before/After指標の記入ヒント：**
> 「作業時間」だけでもOK。**Beforeは今週中に実測**しておくこと。

> 💡 **講師メモ（ノート欄）：**  
> 「⑥⑦⑧は全部完璧でなくてOKです。⑥の実データ移行計画は、Slide 06で説明した3ステップをそのまま自分の業務に当てはめてください。⑦の運用ルールは『誰が・いつ・どこに保存するか』の3点だけでも十分です。⑧のBefore/After指標は、作業時間の1項目だけでも構いません。大事なのは、Beforeの数字を今週中に実測しておくことです。」

#### Slide 14｜試作ガイド（Step 3 ── 最後の10分）

**タイトル：** Step 3：Coworkで試作に着手 ── 完成しなくてOK

**大見出し：**

**試作の進め方（3レベル）：**

| レベル | やること | こんな方向け |
|---|---|---|

**コンテキスト蓄積のヒント（オレンジ枠）：**

> 💡 **講師メモ（ノート欄）：**  
> 「最後の10分で試作に入ります。完成させる必要はありません。Lv.1でプロジェクトにタスクを作るだけでも十分です。Lv.2まで行ける方は、ダミーデータで最初の2工程を動かしてみてください。ここでプロジェクトに設計メモを残しておくと、コンテキストが蓄積されて次回の検証がスムーズになります。ref6で紹介した『プロジェクトに知識を溜める』考え方です。」
