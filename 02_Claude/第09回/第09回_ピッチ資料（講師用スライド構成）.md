### B. 複合ワークフローとは（Slide 04〜07）

---

#### Slide 04｜「単体AI業務担当者」と「複合ワークフロー」の違い

**タイトル：** Phase②で作ったもの → Phase③で作るもの

**左右対比レイアウト：**

| | Phase②：単体AI業務担当者 | Phase③：複合ワークフロー |
|---|---|---|
| **構成** | 1つのプロジェクト内にスキル＋参照情報＋スケジュール | 複数のAI業務担当者を工程順につなげる |
| **データの流れ** | 入力 → AI処理 → 出力（1工程で完結） | 工程1の出力 → 工程2の入力 → …と受け渡し |
| **人間の関与** | 実行の指示と最終確認 | 工程の途中で人間が確認・判断・承認する |
| **エラー時** | AIが止まる or 人間が再指示 | 例外処理の分岐がある（AIが停止→報告→人間が判断） |
| **イメージ** | 「優秀な担当者1人」 | 「チームで業務を回す仕組み」 |

**大見出し（下部・オレンジ）：**
> Phase③の目標：「全自動化」ではなく「人間の判断を含む半自動化」

> 💡 **講師メモ（ノート欄）：**  
> 「Phase②で作ったのは、1つの業務を1人で完結させるAI業務担当者でした。Phase③では、その担当者を複数つなげて、工程間でデータを受け渡し、途中で人間が確認するフローを作ります。重要なのは『全自動』を目指さないこと。人間がチェックすべきポイントを意図的に残します。」

---

#### Slide 05｜複合ワークフローの3つの構成要素

**タイトル：** 複合ワークフローを構成する3つの要素

**3カード（大きく横並び・アイコン付き）：**

| 🔄 工程間データ受け渡し | ✋ 承認ゲート | ⚠️ 例外処理分岐 |
|---|---|---|
| 工程1の出力が工程2の入力になる | 人間が確認・承認してから次の工程に進む | 異常値やエラー発生時に別の処理に分岐する |
| 例：配送実績CSV → 集計レポート → 請求データ | 例：「この集計結果で合っていますか？」→ OK / NG | 例：異常値を検出 → AIが停止 → 人間に報告 → 人間が判断 |
| **ポイント：** ファイル名・フォルダ・書式を工程間で揃える | **ポイント：** 承認なしに次工程へ自動で進ませない | **ポイント：** 「想定外が起きたらAIは止まる」が安全の基本 |

**概念図（下部）：**

```
工程1（AI）→ 出力 → [人間が確認・承認] → 工程2（AI）→ 出力
                                                ↓ 異常値の場合
                                          [AIが停止→人間に報告]
                                                ↓ 人間が判断
                                          修正して再実行 or 手動対応
```

> 💡 **講師メモ（ノート欄）：**  
> 「複合ワークフローは3つの要素でできています。①工程間のデータ受け渡し、②人間の承認ゲート、③例外処理の分岐です。特に②と③が重要です。AIに任せっぱなしにせず、要所で人間が確認する。想定外のことが起きたらAIは自動で止まって報告する。これが安全な半自動化の基本です。」

**タイトル：** 具体例：配送実績 → 異常検出 → 請求更新（物流企業の月次フロー）

**フロー図（縦方向・6ステップ・大きく表示）：**

```
┌─────────────────────────────────────┐
│ 工程1：AI配車アシスタントが配送実績を集計          │  ← AI
│ 入力：配送実績CSV  出力：月次集計レポート          │
└─────────────┬───────────────────────┘
              ↓
┌─────────────────────────────────────┐
│ 工程2：AIが異常値を検出                          │  ← AI
│ 「遅延率が前月比150%」「A社の単価が通常と異なる」  │
└─────────────┬───────────────────────┘
              ↓
┌─────────────────────────────────────┐
│ ✋ 承認ゲート：人間が確認・判断                   │  ← 人間
│ 「この異常値は季節要因か？ → 問題なし / 要調査」    │
└──────┬──────────────┬────────────────┘
    OK ↓            NG ↓
    通常処理     ⚠️ 例外処理
       ↓         （人間が調査・修正）
┌─────────────────────────────────────┐
│ 工程3：AI経理アシスタントが請求データを更新         │  ← AI
│ 入力：承認済み集計  出力：請求書ドラフト            │
└─────────────┬───────────────────────┘
              ↓
┌─────────────────────────────────────┐
│ ✋ 最終承認：人間が請求書を確認・送付               │  ← 人間
└─────────────────────────────────────┘
```

**補足テキスト（下部）：**
> 💡 AI が処理する工程と、人間が判断する工程が交互に入る「半自動化」のフロー

> 💡 **講師メモ（ノート欄）：**  
> 「物流企業の月次業務を例にします。まずAIが配送実績を集計し、次にAIが異常値を検出します。ここで人間が『この異常値は本当に問題か？』を判断します。問題なければAIが請求データを更新し、最後に人間が請求書を確認して送付します。AIが処理する工程と人間が判断する工程が交互に入っている。これが複合ワークフローの基本形です。」

#### Slide 07｜安全設計の原則

**タイトル：** 安全設計の原則：「AIが判断してよい範囲」と「人間が判断すべき箇所」

**2列対比（大きく表示）：**

| ✅ AIに任せてよいこと | ✋ 人間が判断すべきこと |
|---|---|
| データの集計・整形・フォーマット変換 | 金額の最終確認・承認 |
| 定型文の生成（メール文面・報告書ドラフト） | 異常値の原因判断（季節要因？ミス？） |
| ファイルの検索・振り分け・名前変更 | 顧客への対外送信の承認 |
| 既知パターンのチェック（書式・必須項目の有無） | 例外的なケースへの方針決定 |
| ルールが明確な判断（閾値超え→フラグを立てる） | 担当者の変更・業務フローの変更 |

**原則（3つ・オレンジ枠）：**

> **原則①** 「お金が動く」「社外に出る」工程の手前には、必ず人間の承認ゲートを置く  
> **原則②** AIが想定外の状況に遭遇したら、自動で止まって人間に報告する設計にする  
> **原則③** 「AIが100%正しいとは限らない」前提で、チェックポイントを設ける

> 💡 **講師メモ（ノート欄）：**  
> 「この表を覚えておいてください。AIに任せてよいのは定型的な処理です。金額の確認、異常値の判断、対外送信の承認は人間がやるべきです。原則は3つ。①お金・対外の手前には承認を入れる、②想定外ならAIは止まる、③AIは100%ではない前提で設計する。次回の設計回でこの考え方を使って自分のワークフローを設計します。」

#### Slide 20｜よくある質問 Q&A

**タイトル：** よくある質問

**Q&A カード形式：**

**Q1：「複合ワークフローは難しそうです。Phase②のAI業務担当者だけでも十分では？」**
> A：Phase②の成果だけでも十分に実務で使えます。ただし、前後の工程と手動でつないでいる部分があれば、そこを複合ワークフロー化すると**さらに時間を削減**できます。無理に複雑にする必要はありません。「いま手作業でつないでいる部分を1つだけ自動化する」くらいのスモールスタートがおすすめです。

**Q2：「承認ゲートを入れると、結局人間がやることが増えませんか？」**
> A：承認ゲートは**確認するだけ**です。データの集計やドラフト作成はAIがやっているので、人間は「結果を見てOKかNGかを判断する」だけ。判断に要する時間は**数分**で、手作業で全工程をやるよりはるかに短縮されます。

**Q3：「例外処理はどこまで設計すればいいですか？」**
> A：最初から全パターンを想定する必要はありません。まずは**「AIが想定外の状況に遭遇したら止まって報告する」**というルールだけ入れておけば安全です。運用しながら、頻出する例外パターンを1つずつ追加していけばOKです。

> 💡 **講師メモ（ノート欄）：**  
> 「質問を受け付けます。よくある質問を3つ紹介します。Q1は『難しそう』という不安。Phase②だけでも使えますが、手作業でつないでいる部分が1つでもあれば、そこだけ複合化するのがおすすめです。Q2は承認ゲートの手間。確認するだけなので手間は小さいです。Q3は例外処理の範囲。最初は『止まって報告する』だけでOKです。」
