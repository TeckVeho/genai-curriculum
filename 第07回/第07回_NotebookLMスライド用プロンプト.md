# 第07回 NotebookLMスライド用プロンプト

> **使い方:** 以下の「---」以降のテキストをすべてコピーし、NotebookLMの「追加の指示」欄に貼り付けてください。
> ソースとして `materials/brand-colors.md` と `materials/background.png` をアップロード済みであることを確認してください。

---

## スライド仕様

- スライド枚数: 20枚
- 言語: 日本語
- 1スライド1メッセージ（テキストは最大5〜6行）
- 全スライドにプログレスバー（セクション名 + スライドNo.）を固定ヘッダーとして表示

## ブランドカラー（厳守）

以下のカラーのみを使用してください。他の色は使わないでください。

- メインカラー（青）: #005BAC → 見出し背景、プログレスバー、ステップ番号の丸
- サブカラー（濃紺）: #02396B → ヘッダー・フッター背景
- アクセントカラー（オレンジ）: #F58220 → CTA、ハイライト、重要メッセージ枠
- ベースカラー: #F5F5F5 → スライド背景
- テキスト（濃色背景上）: #FFFFFF
- テキスト（明色背景上）: #171717

## 背景画像

- 添付の background.png を全スライドの背景として使用してください
- 背景はスライドごとに変えず、全スライドで統一してください

## ロゴスペース

- 右上（約 120×40px 相当）にロゴ配置用のスペースを確保してください
- このスペースにはコンテンツを配置しないでください
- 白い矩形で囲わず、背景に馴染ませてください

## フォント

- 見出し: Noto Sans JP Bold（36〜48pt相当）
- 本文: Noto Sans JP Regular（18〜22pt相当）

## トーン

- DX研究所のブランドらしく、プロフェッショナルかつ親しみやすい
- 専門用語は噛み砕いた補足を添える
- 物流事務の現場担当者が直感的に理解できる表現

## スライド構成

### スライド 1: タイトルスライド
- 大見出し: 第07回：[ワークショップ] ソリューション構築 (1) — プロトタイプ開発 ── まず動くものを1つ作る
- サブテキスト: 物流事務DX人材育成プログラム「AI活用・業務変革コース（全10回）」
- 掲載要素: 研修ロゴ / 自社グループ名 / 実施日 / 講師名（右下）
- デザイン: Codex Webの画面キャプチャ（ぼかし加工）を背景にネイビーのオーバーレイ。白テキストで大見出しを中央配置。「実践」フェーズバッジをオレンジで右上に配置。

### スライド 2: ここまでの道のり（第01〜06回）
- フロー図（横並び、第07回をハイライト）:
  - 第01回「業務棚卸し」
  - 第02回「ChatGPT概論」
  - 第03〜05回「座学（Skills / Codex を学ぶ）」
  - 第06回「ソリューション設計」
  - 第07回「プロトタイプ開発」← 今ここ（オレンジで強調）
  - 第08〜09回「改善・発表準備」
  - 第10回「成果発表会」
- 強調テキスト: 前回（第06回）で設計書を作りました。今日はそれを実際に動くものにします。

### スライド 3: 本日のアジェンダ
- タイムテーブル形式:
  - 00〜05分: 本日のゴールと進め方の確認
  - 05〜50分: 構築ワーク（Codex組 / Skills組 並行・45分）
  - 50〜55分: 進捗共有 & 次回への引き継ぎ
  - 55〜58分: フィードバックアンケート
  - 58〜60分: まとめ・クロージング
- デザイン: 縦タイムラインまたはステップフロー形式。「構築ワーク（45分）」の行をオレンジで強調（本日の最重要パート）。

### スライド 4: 本日のゴール
- タイトル: 今日のゴール：「まず動くものを1つ作る」
- 大見出し（インパクト重視）: 完璧を目指さない。動く・使える最低限の状態を作る。
- 4つの到達目標カード（横並び）:
  - ① 指示文に変換: 設計書をAIへの指示文に変換しCodex/Skillsに投入できる
  - ② プロトタイプ完成: 最小動作版のプロトタイプを1つ完成させられる
  - ③ 修正指示を出せる: 思い通りに動かないときに修正指示を出せる
  - ④ 改善点の書き出し: 次回で改善すべき点を3つ以上書き出せる

### スライド 5: Codex組 / Skills組の構築フロー
- タイトル: 2つの組に分かれて並行作業（45分）
- 2カラム比較表:
  - Codex組: Step1（10分）設計書→指示文に変換 → Step2（15分）Codexに投入して実行 → Step3（10分）動作確認と修正指示 → 成果物：動くWebアプリ
  - Skills組: Step1（10分）定義書→指示文を完成 → Step2（15分）Skill作成 → Step3（10分）テスト実行と修正 → 成果物：テスト完了した基本Skill
- 補足テキスト: 残り10分は予備時間です。早く終わった方は修正・改善を進めてください。

### スライド 6: 【Codex組】Step 1 — 設計書を指示文に変換する
- タイトル: 【Codex組】Step 1：設計書を指示文に変換しよう（10分）
- 大見出し: 設計書の内容を、Codexに渡せる形の日本語指示文に書き直す
- テンプレート（枠で囲んで大きく表示）:
  - 「シンプルなWebアプリを作ってください。アプリ名：[アプリ名]」
  - 入力フォームの項目：[項目1]（テキスト/数字/日付等）、[項目2]、[項目3]
  - 送信ボタンを押すと入力した内容をフォーム下の表に追記表示する
  - 表のヘッダー：[項目1]・[項目2]・[項目3]
  - データはブラウザのローカルストレージに保存（ページ更新後も消えない）
- ポイント（オレンジ枠）: テンプレートの [ ] 部分を自分の設計書の内容に置き換えるだけでOK

### スライド 7: 【Codex組】Step 2 — Codexに投入する
- タイトル: 【Codex組】Step 2：Codexに投入しよう（15分）
- 5ステップフロー（横並び）:
  - ① New Taskを押す → ② 指示文を貼り付けて送信 → ③ 実行ログを観察しながら待つ → ④ Changesタブで確認 → ⑤ Applyして動作確認
- 重要ポイント（オレンジ枠）: Codexの処理中は画面を閉じないでください。処理が終わるまで1〜5分かかることがあります。

### スライド 8: 【Codex組】Step 3 — 動作確認 & 修正指示
- タイトル: 【Codex組】Step 3：動作確認 & 修正指示（10分）
- 確認チェックリスト（3項目・大きく表示）:
  - □ フォームの全項目が表示されている
  - □ 送信ボタンを押すと一覧に追記される
  - □ ページを更新してもデータが残る
- 修正指示の出し方（問題がある場合）:
  - ボタンが表示されない → 「送信ボタンが見当たりません。追加してください」
  - データが保存されない → 「ローカルストレージに保存されるように修正してください」
  - レイアウトが崩れている → 「フォームと一覧表のレイアウトを整えてください」
- 補足: 1回で完璧にならなくて当然。修正指示を出すのもCodexの使い方です。

### スライド 9: 【Skills組】Step 1 — 定義書をもとに指示文を仕上げる
- タイトル: 【Skills組】Step 1：定義書をもとに指示文を仕上げよう（10分）
- 大見出し: 第06回の定義書の「指示文の下書き」を確認・補強する
- 3つのチェック観点カード（横並び）:
  - 役割は明記されているか: 例「あなたは物流会社のカスタマーサポート担当です」
  - 出力形式は具体的か: 例「300文字以内・です/ます調・件名→本文→締めの構成」
  - 入力形式を含めているか: 例「ユーザーは相手先名・理由・日付の3つを入力します」
- ポイント（オレンジ枠）: 指示文に「具体的な条件」が多いほど、Skillの出力は安定します

### スライド 10: 【Skills組】Step 2 — Skillを作成する
- タイトル: 【Skills組】Step 2：Skillを作成しよう（15分）
- 手順（番号付きリスト・大きく表示）:
  1. ChatGPTのSkills画面を開き「新しいSkillを作成」
  2. Skill名を入力（定義書の名称をそのまま使ってOK）
  3. 説明を入力（このSkillが何をするか一文で）
  4. Instructions（指示文）にStep 1で仕上げた指示文を貼り付け
  5. 保存する
- 補足: Skill名と説明は後から変更できます。まずは保存を優先してください。

### スライド 11: 【Skills組】Step 3 — テスト実行 & 修正
- タイトル: 【Skills組】Step 3：テスト実行 & 修正（10分）
- 確認チェックリスト（3項目・大きく表示）:
  - □ 期待通りの出力が返ってきている
  - □ 文体・文字数・含めるべき要素が揃っている
  - □ 想定外の出力がない（または許容範囲内）
- 修正指示の出し方:
  - 出力が長すぎる → 「300文字以内で出力してください」を追加
  - 文体が合わない → 「です/ます調で統一してください」を追加
  - 必要な要素が抜ける → 「必ず①件名 ②本文 ③締めの一文を含めてください」を追加
- テスト方法: 定義書に書いたテスト用入力例を使って実行 → 出力を確認 → 問題があればInstructionsを修正 → 再テスト

### スライド 12: ワーク中タイマースライド
- タイトル: 構築ワーク中… ── まず動くものを1つ作ろう
- 画面の大部分にタイマー表示（残り時間を大きく）
- 最小限のガイドテキスト（下部に2カラム）:
  - Codex組: ①設計書→指示文に変換（10分）→ ②Codexに投入→Apply（15分）→ ③動作確認→修正指示（10分）
  - Skills組: ①定義書→指示文を完成（10分）→ ②Skillを作成→保存（15分）→ ③テスト実行→修正（10分）
- ステータス: 🟢順調→そのまま進めてください / 🟡エラー→手を挙げてください / 🔵完了→修正・改善を進めてください
- デザイン: 白背景 + 大型カウントダウン表示。作業に集中できるようシンプルに。

### スライド 13: 講師巡回ポイント（講師用）
- タイトル: 【講師用】巡回時のチェックポイント
- 巡回ポイント（4項目・Q&A形式）:
  - 「Codexが動かない」→ 指示文が曖昧でないか確認。具体的な項目名・動作を追加するよう促す
  - 「Skillの出力がおかしい」→ Instructionsの条件を1つずつ確認し、足りていない制約を追加
  - 「何から手をつければいいかわからない」→ 「まず設計書の1番目の項目だけでいいです。入力フォームだけ / Skill名だけから始めましょう」
  - 環境接続に問題がある → トラブルシューティング参考資料を参照
- 巡回タイミングの目安: ①開始5分後 ②開始15分後 ③開始30分後 ④開始40分後

### スライド 14: 進捗共有
- タイトル: 今日の進捗を報告してください（5分）
- 話すこと（3点・大きなカード横並び）:
  - ① 作ったもの: 一言で何を作ったか
  - ② 動いた / 動かなかった: 現時点の動作状況
  - ③ 次回改善したいこと: 次回取り組みたいこと
- 進行: 全員に一言ずつ聞きます（1人30秒〜1分）

### スライド 15: 次回への引き継ぎ
- タイトル: 次回（第08回）に向けた準備
- 必須アクション（2つ）:
  - ① 今日の成果を保存: Codex組→GitHubリポジトリにコード保存確認 / Skills組→ChatGPT SkillsにSkill保存確認
  - ② 改善したいことを3つ書き出す: 配布資料の「次回改善リスト」に記入
- 次回予告（オレンジ枠）: 次回：第08回「ソリューション構築 (2) ── 機能追加・改善」。今日作ったプロトタイプに機能を追加・改善します。「ここをこうしたい」と思ったことをメモしておいてください。

### スライド 16: よくある質問 Q&A
- タイトル: よくある質問
- Q1:「Codexがエラーを返してきました。どうすればいいですか？」→ A: エラーメッセージをそのままCodexに渡して「このエラーを解決してください」と指示してください。Codexは自分が出したエラーを自分で直すのが得意です。
- Q2:「プロトタイプが完成しませんでした」→ A: 大丈夫です。次回（第08回）の冒頭で追いつく時間を設けます。今日の作業内容と、どこで詰まったかをメモしておいてください。
- Q3:「Skillsの出力が毎回少し違います。安定させるには？」→ A: Instructionsに出力のパターンを具体的に書くと安定します。例：「必ず以下の構成で出力：①件名 ②本文（3段落） ③締めの一文」

### スライド 17: フィードバックアンケート
- タイトル: 今日のフィードバックにご回答ください（3分）
- 内容: 全5問・約3分・正解はありません。理解度 / 業務活用イメージ / 進行スピード / 満足度 / 自由記述
- デザイン: QRコードまたは短縮URLを中央に大きく表示

### スライド 18: 本日のまとめ
- タイトル: 本日のまとめ
- 到達目標チェック:
  - □ 設計書をAIへの指示文に変換し、Codex / Skills に投入できた
  - □ プロトタイプ（最小動作版）を1つ完成させた（または着手できた）
  - □ 思い通りに動かないときに修正指示を出せた
  - □ 次回で改善すべき点を3つ以上書き出せた
- 重要メッセージ: 今日のプロトタイプ、よく作れました。次回はこれに機能を追加・改善します。
- 保存確認: Codex組→GitHubリポジトリに保存済みですか？ / Skills組→ChatGPT SkillsにSkill保存済みですか？

### スライド 19: 【配布資料】プロトタイプ構築チェックリスト
- タイトル: プロトタイプ構築チェックリスト
- Codex組チェックリスト:
  1. Codexにタスクを投入し、コードが生成された
  2. Changesタブで内容を確認した
  3. Applyしてブラウザでアプリを開いた
  4. フォームの全項目が表示されている
  5. 送信ボタンを押すと一覧に追記される
  6. GitHubリポジトリにコードが保存されている
- Skills組チェックリスト:
  1. Skill名・説明・Instructionsを入力した
  2. テスト用の入力例で実行した
  3. 出力の文体・文字数・要素が期待通りか確認した
  4. 問題があればInstructionsを修正した
  5. Skillが保存されている
- デザイン: 印刷用に白背景・黒枠テーブル。A4縦向き1枚に収まるサイズ。Codex組・Skills組を上下2段で配置。

### スライド 20: 【配布資料】次回改善リスト
- タイトル: 次回（第08回）で改善したいこと
- 改善リスト記入欄（5行分）: 各行に「改善したい内容」と「優先度（高/中/低）」の記入スペース
- 記入のヒント:
  - 「こんな機能を追加したい」（例：CSVエクスポート、デザイン改善）
  - 「ここが動かなかった」（例：保存が効かない、出力が不安定）
  - 「もっとこうしたい」（例：入力項目を増やしたい、文体を変えたい）
- 次回のアクション: このリストを持参してください。次回はこの改善リストをもとに作業します。
