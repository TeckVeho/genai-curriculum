# __第08回：[ワークショップ] ソリューション構築 (2)：改善・機能追加__

---

## __1\. 講義概要__

- __フェーズ:__ 実践
- __所要時間:__ 60分
- __形式:__ オフライン

### __狙い__

第07回で作ったプロトタイプを**実用レベルに引き上げる**。Codex 組はUIの改善・機能追加・エッジケース対応を、Skills 組は入力パターンの網羅・出力品質の安定化を行う。「段階的に指示を重ねてソリューションを育てる」感覚を定着させる。終了時に**簡易アンケート**を実施する。

### __到達目標__

1. 前回の「改善したいことリスト」を2つ以上反映できる
2. 「うまくいかなかった指示」を「うまくいく指示」に修正する経験を積める
3. 次回（仕上げ）に向けて残り作業を整理できる
4. 「自分の業務で実際に使えそうか」を具体的にイメージできる

### __準備物（講師）__

- PC（GitHub・ChatGPT Plus・Codex Web を開いておく）
- 前回の各受講者の進捗メモ（第07回の終わりに確認した内容）

### __準備物（受講者）__

- PC（GitHub・ChatGPT アカウント利用可能な状態）
- 第07回で作ったプロトタイプ（Codex 組：GitHub に保存済み / Skills 組：ChatGPT に保存済み）
- 第07回テイクアウェイの「改善したいことリスト」

### __【08\-50分】 改善・機能追加ワーク（Codex 組 / Skills 組 並行）__

- __フェーズ:__ 個人作業（講師は両組を巡回）

---

#### __Codex 組：アプリの改善・機能追加__

以下のメニューから「改善したいことリスト」に合わせて選択し、Codexに指示を出す。

**改善メニュー（例）:**

| カテゴリ | 改善内容の例 | Codex への指示例 |
| --- | --- | --- |
| UI改善 | 色・レイアウトを見やすくする | 「テーブルのヘッダー背景を紺色、文字を白にしてください」 |
| 機能追加 | 削除ボタンをつける | 「各行に削除ボタンを追加してください。削除したらその行がなくなるようにしてください」 |
| 機能追加 | 検索・絞り込みをつける | 「テキスト検索ボックスを追加し、業務内容で絞り込めるようにしてください」 |
| 機能追加 | CSV ダウンロードをつける | 「一覧データをCSVとしてダウンロードできるボタンを追加してください」 |
| バグ修正 | データが消えてしまう | 「ページ更新後もデータが消えない実装になっているか確認してください。消える場合は修正してください」 |
| 機能追加 | 完了フラグをつける | 「各行に『完了』チェックボックスを追加し、チェックした行はグレーアウトするようにしてください」 |

**作業の進め方:**

1. 改善したいことを1つ選んで指示文を作成
2. Codex に投入 → Changes タブで確認 → Apply
3. ブラウザで動作確認
4. 問題があれば修正指示を追加
5. 次の改善項目に移る（時間まで繰り返す）

---

#### __Skills 組：入力パターンの網羅・出力品質の安定化__

**改善メニュー（例）:**

| カテゴリ | 改善内容の例 | Instructions の修正方向 |
| --- | --- | --- |
| 出力形式の安定 | 毎回構成が変わる | 出力の構造（①件名 ②本文3段落 ③締め）を Instructions に明記 |
| 入力パターンの追加 | 入力が少ない時の対応 | 「情報が少ない場合は『確認が必要な項目』として出力に含める」と追記 |
| トーンの調整 | 少し硬すぎる / 柔らかすぎる | 「ビジネス文体だが親しみやすい口調で」など表現を調整 |
| エッジケース対応 | 遅延日数が0日の場合 | 「遅延日数が0の場合は『現時点では予定通りです』と書く」と条件を追加 |
| 多パターン出力 | 3案出してほしい | 「3つのバリエーションを出力してください」と追記 |

**作業の進め方:**

1. テスト入力を3〜5パターン用意し、それぞれ実行して出力を確認
2. 問題があった入力パターンについて、Instructions に対処するルールを追加
3. 再テストして出力が安定したことを確認
4. 次のパターンへ

- __巡回時の共通指導ポイント:__
	- 「1つ直したら必ずテストして確認してから次へ。複数を一度に変えると何が効いたかわからなくなります」
	- 「Codexが意図と違う変更をした場合は『〇〇は変えないでください。△△だけ直してください』と限定的に指示を出してください」
	- 「行き詰まったら『現状の問題点を教えてください』とAIに聞いてみましょう」

## __3\. 想定される質問と回答 \(Q&A\)__

- __Q1: 「改善したいことが多すぎて、何から手をつけるかわかりません」__
	- __A:__ 「まず『使えない理由になるもの』から直してください。例：フォームが動かない・出力が全然違う。見た目の改善は後回しでOKです。」

- __Q2: 「Codex が同じ指示を出しても違う結果を返してくることがあります」__
	- __A:__ 「AIの特性上、完全に同じ出力を保証することは難しいです。ただし指示が具体的であるほど安定します。『〇〇のファイルの△△行目を変更してください』のように、変更箇所を限定する指示を心がけてください。」

- __Q3: 「Skills の出力が長くなりすぎます」__
	- __A:__ 「Instructions に『〇〇文字以内』または『3行以内』と文字数・行数の制約を追加してください。それでも長い場合は、出力の各パーツに個別に文字数制限をつけるのが効果的です。例：『件名は30文字以内、本文は150文字以内』。」
