**タイトル：** プラグイン＝ Codex と外部サービスをつなぐ「橋」

**デザイン提案：** Codexアイコンを中央に、周囲に各サービスのロゴアイコンを放射状に配置。接続線をブルーのアニメーション風に表現。

#### Slide 08｜OAuth認証でつまずいたときの対処法

**タイトル：** 接続がうまくいかないときは ── 3つのチェックポイント

**トラブルシューティングカード（3枚・縦並び・大きく表示）：**

| # | つまずき | 対処法 |
|---|---|---|
| ① | **プラグインが見つからない** | 検索ワードを **英語** で試す（例：「メール」→「gmail」）。大文字・小文字は関係ない |
| ② | **OAuth認証画面が出ない** | ブラウザの **ポップアップブロック** を確認。Chrome の場合：アドレスバー右のブロックアイコン → 「常に許可」をクリック |
| ③ | **「権限がありません」と表示される** | 会社の Google アカウントではなく **個人アカウント** でログインしている可能性あり。正しいアカウントか確認 |

**ポップアップブロック解除の手順（Chrome の場合）：**

```
1. アドレスバー右側の「🚫ブロック」アイコンをクリック
2. 「ポップアップを常に許可する」を選択
3. もう一度プラグインの「インストール」をクリック
```

**デザイン提案：** 各トラブルシューティング項目を赤・黄・緑の信号色で分類。ポップアップブロック解除の手順は Chromeのスクリーンショット付きで説明。

> 💡 **講師メモ（ノート欄）：**  
> 「OAuth認証が最大のつまずきポイントです。ポップアップブロックが原因で認証画面が出ないケースが最も多いです。巡回時に『認証画面が出ません』と言われたら、まずブラウザのポップアップブロックを確認してください。Chromeであればアドレスバー右にブロックアイコンが出ているはずです。」

#### Slide 10｜単発タスク実行とは？

**タイトル：** 今日のゴール：「1回だけ動かす」

**概念図（大きく中央に）：**

```
┌─────────────────────┐
│  自動化仕様カード     │
│  （前回作成）         │
│                     │
│  「未返信メールを     │
│   取得してSlackに     │
│   通知して」         │
└─────────┬───────────┘
          ↓  指示
┌─────────────────────┐
│  Codex              │
│  + プラグイン         │
│  （さっき接続した）   │
└─────────┬───────────┘
          ↓  実行
┌─────────────────────┐
│  結果を確認           │
│  「ちゃんと動いた！」  │
└─────────────────────┘
```

**重要メッセージ（オレンジ枠で強調）：**
> **今日は「1回動かす」だけでOK。**  
> 毎日自動で実行する「定期実行（オートメーション）」は **次回** やります。  
> まずは1回動かして「ちゃんと期待通りの結果が出るか？」を確認しましょう。

> 💡 **講師メモ（ノート欄）：**  
> 「大事なポイントです。今日は1回動かすだけです。毎朝自動で動くようにする『定期実行』は次回やります。なぜ分けるかというと、まず1回動かして結果を確認しないと、定期実行にしたときに毎回おかしな結果が出続けてしまうからです。料理で言えば、味見をしてから量産する、という順番です。」

#### Slide 11｜指示文の組み立て方

**タイトル：** Codex への指示文＝仕様カードの内容を「話し言葉」で伝えるだけ

**指示文の3要素テンプレート（大きく表示）：**

```
① 何をしてほしいか（タスク）
② どこから / どこへ（入力元 → 出力先）
③ どんな形式で（出力形式の指定）
```

**具体例3パターン：**

| 仕様カードの内容 | Codex への指示文 |
|---|---|
| 未返信メール → 重要度付きでSlack通知 | 「過去3日間の未返信メールを **Gmail** から取得して、重要度を付けて **Slack** の #重要メール チャンネルに通知して」 |
| 物流ニュース → 要約してSlack投稿 | 「今日の物流業界ニュースを5件検索して、各100文字以内で要約して **Slack** の #ニュース チャンネルに **日本語で** 投稿して」 |
| Google Drive の最新ファイル → 要約 | 「**Google Drive** の『月次報告』フォルダにある最新ファイルの内容を **箇条書き5点** で要約して」 |

**ポイント（下部に強調）：**
- プラグイン名（Gmail、Slack など）を指示文に含めると、Codex がどのプラグインを使うか迷わない
- 出力の言語・形式・文字数を明示すると精度が上がる

> 💡 **講師メモ（ノート欄）：**  
> 「指示文は難しく考えなくて大丈夫です。仕様カードに書いた内容を、話し言葉でCodexに伝えるだけです。コツは3つ。①何をしてほしいか、②どこから取ってどこに出すか、③どんな形式で出すか。この3つを書けば、Codexは理解してくれます。例文をそのまま参考にしてもOKです。」

#### Slide 12｜【講師デモ】単発タスクを実行してみよう

**タイトル：** 実行デモ ── 「未返信メールをSlackに通知」を1回実行

**デモの手順（5ステップ・スクリーンショット付き）：**

| Step | 操作 | 確認ポイント |
|---|---|---|
| ① | Codex のチャット画面を開く | プラグイン接続済みのアイコンが表示されているか |
| ② | 指示文を入力して送信 | 「過去3日間の未返信メールをGmailから取得して、重要度を付けてSlackの #テスト チャンネルに通知して」 |
| ③ | Codex が処理を開始（ログを観察） | 「Gmail にアクセスしています…」「Slack に投稿しています…」と進行状況が出る |
| ④ | Slack を開いて結果を確認 | 指定したチャンネルに投稿されているか |
| ⑤ | 結果を評価 | 期待通りの内容か？足りない情報はないか？ |

**結果が期待と違った場合：**
> 指示文をもう少し具体的にして、もう一度実行するだけでOK  
> 例：「重要度は【高】【中】【低】の3段階で付けて」「送信者名も含めて」

**デザイン提案：** 左半分にCodex画面のスクリーンショット、右半分にSlackの通知結果のスクリーンショットを並べて表示。成功の矢印で結びつける。

> 💡 **講師メモ（ノート欄）：**  
> 「これから実際に実行します。私の画面を見ていてください。…（実演）。Slackを開いてみましょう。…はい、投稿されていますね。もし結果が期待と違った場合は、指示文をもう少し具体的にしてもう1回実行するだけです。何度でもやり直せますので安心してください。」

#### Slide 13｜単発タスク実行ワークの説明

**タイトル：** 自分の仕様カードの内容を Codex で実行しよう（20分）

**やること（4ステップ）：**

| Step | 内容 | 目安時間 |
|---|---|---|
| ① | 自動化仕様カードを見て、指示文を組み立てる | 5分 |
| ② | Codex に指示文を入力して実行 | 3分 |

| ④ | 期待通りでなければ指示文を修正して再実行 | 7分 |

**指示文を書くときのチェックリスト：**

| チェック | 内容 |
|---|---|
| □ | **何をしてほしいか** を書いたか？ |
| □ | **プラグイン名**（Gmail / Slack 等）を明記したか？ |
| □ | **出力先**（どのチャンネル / フォルダ）を指定したか？ |
| □ | **出力形式**（日本語・箇条書き・文字数等）を指定したか？ |

**タイマー表示：** ⏱ 20分（大きく中央に表示）

> 💡 **講師メモ（ノート欄）：**  
> 「20分間でStep①〜④に取り組んでください。まず仕様カードを見ながら指示文を組み立てて、Codexに入力して実行します。結果を確認して、期待通りでなければ修正してもう一度。何度でもやり直してOKです。手が止まったら声をかけてください。」

#### Slide 14｜単発タスク実行ワーク中タイマースライド

**タイトル：** 単発タスク実行ワーク中…

**画面の大部分：** タイマー表示（残り時間を大きく）

**最小限のガイドテキスト（下部）：**

| 進捗 | ステータス |
|---|---|
| 指示文を書いている | 🔵 作業中 |
| 1回目を実行した | 🟡 結果確認中 |
| 結果が期待通りだった | 🟢 完了！ |
| 修正して再実行中 | 🔵 改善中（何度でもOK） |

**よくあるつまずきと対処法（右下に常時表示）：**

| つまずき | 対処法 |
|---|---|
| Slack通知が届かない | チャンネル名が正確か確認（「#」は不要） |
| 結果が英語で返ってくる | 指示文に「日本語で」を追加 |
| 結果が期待と違う | 出力形式をより具体的に指定（例：「箇条書きで」「表形式で」） |
| 「プラグインが見つかりません」 | プラグイン一覧（Slide 06）を再確認。接続済みか確認 |

**デザイン提案：** 白背景 + 大型カウントダウン表示。つまずき対処法を右下に常時表示。

> 💡 **講師メモ（ノート欄）：**  
> 積極的に巡回する。5分経過時点で「まだ指示文が書けていない方は手を挙げて」と声をかける。10分経過時点で「1回目を実行できた方は？」と確認。Slack通知系のタスクで「チャンネルが見つからない」エラーが出ている受講者には、チャンネル名に「#」を付けていないか確認する（不要）。結果が期待通りでなくても「修正して再実行するだけでOK」と安心させる。

#### Slide 16｜よくある質問 Q&A

**タイトル：** よくある質問

**Q&A カード形式：**

**Q1：「実行結果が期待通りではないのですが、大丈夫でしょうか？」**
> A：まったく問題ありません。**指示文を修正して再実行する** ことで精度が上がります。1回で完璧になることはほぼありません。次回以降も改善を続けていきます。

**Q2：「プラグインが接続できませんでした。次回までに準備すべきことは？」**
> A：講師に個別相談してください。ブラウザの設定やアカウントの問題である場合が多いです。次回の冒頭でフォローアップの時間を設けます。

**Q3：「定期実行を早く試してみたいのですが、今日やってもいいですか？」**
> A：意欲的ですね！ただし、**まず単発実行の結果が安定してからの方が安全** です。次回の第07回で全員一緒に定期実行を設定しますので、今日は「結果の品質を上げること」に集中してください。
